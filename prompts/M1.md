# M1 — Foundation

You are building FeltSide, a poker game engine, from scratch. This file is
your complete brief for milestone M1. Work strictly inside this repo.

## Step 0 — Bootstrap (one-time)
1. `git init` this directory if not already a repo; create a **private**
   GitHub repo named `FeltSide` with `gh repo create` if `gh` is
   authenticated — otherwise stay local and tell Jo to create/push later.
2. Read `README.md`, then `docs/PRD.md`, then EVERY file in `docs/specs/`.
   These are binding. Spec beats code beats your preference. If a spec is
   silent on something you need, STOP and ask Jo — never guess.
3. Import legacy modules per `legacy/LEGACY_MANIFEST.md` (the old
   poker-trainer repo is on this machine; ask Jo for its path only if you
   can't find it). Copy ONLY the listed files. Do not read anything else
   in the old repo. Commit as `chore: import legacy manifest modules`.

## M1 Scope (PRD §11 M1)
Foundation only: database schema core, auth, bankroll ledger, thin lobby
shell. **The engine does not play poker yet.**

### 1. Project scaffold
- `server/`: Node (ESM) + Express + socket.io installed but sockets used
  only for a trivial connection ping. Supabase JS client (service role).
- `client/`: React + Vite + Tailwind. English UI (PRD §12).
- Test runner on both sides; CI-runnable `npm test` at root.

### 2. Database schema (Supabase migrations, append-only)
- `players`: id (uuid), display_name, email (unique), password_hash,
  role ('coach' | 'player'), crm_student_id (nullable text),
  must_change_password (bool), status ('active' | 'archived'),
  owner_coach_id (threaded per PRD §1), timestamps.
- `bankroll_accounts` + `bankroll_transactions` exactly per
  `docs/specs/RUNTIME.md` §5, including the atomic apply-transaction
  Postgres function (balance update + insert in one transaction,
  CHECK balance >= 0) and the auditable invariant.
- `tables`: id, mode ('coached_cash' | 'uncoached_cash' | 'tournament'),
  status ('scheduled' | 'open' | 'active' | 'completed'), created_by,
  config (jsonb), crm_entry_id (nullable), timestamps. Lifecycle beyond
  listing is M2+.

### 3. Auth (engine owns it — PRD §2)
- Coach account seeded from env (email + initial password) on first boot.
- Endpoints: login (JWT), change-password (forced when
  must_change_password), me, coach-only: create player, archive player,
  set/clear crm_student_id, reset a player's password.
- No self-registration, no email flows, no password reset self-service.
- ONE enforcement path: a single requireAuth + requireCoach middleware
  pair used everywhere. No parallel permission systems (PRD §2 anti-goal).

### 4. Bankroll ledger API
- Coach: adjust (delta or reset-to-X computed as delta, with note),
  view any player's balance + transaction log.
- Player: view own balance + log.
- Buy-in/cash-out transaction types exist in the enum but have no
  triggering flows yet (M2).

### 5. Thin lobby shell (client)
- Login screen incl. forced password change.
- Lobby: lists tables from the API (empty state fine), shows own bankroll
  balance, coach sees an admin drawer: player list, create player,
  bankroll adjust.
- No table pages, no game UI.

## Acceptance criteria (all must pass)
1. Fresh clone + env → migrations apply, coach seeded, server + client
   boot.
2. Coach creates a player; player logs in; forced password change works;
   player cannot reach coach endpoints (401/403).
3. Bankroll: adjustments apply atomically; a test proves
   balance == sum(transactions) after concurrent adjustments; negative
   balance is impossible.
4. Lobby renders; every API endpoint rejects unauthenticated requests.
5. `npm test` green; zero imports from `legacy/` anywhere in server/ or
   client/ (legacy modules graduate in later milestones, not now).

## Explicitly OUT of M1 (do not build)
Game loop, dealing, hands, sockets beyond ping, scenarios, analyzers,
tournaments, export API, lesson sync, review. Any file for these is scope
violation.

## Discipline
- Deviations from this prompt or the specs → one entry each in
  `docs/decisions/` with a one-line justification.
- Unresolved question → stop and ask Jo. Never invent product behavior.
- Migrations append-only; every endpoint authed; no console.log debris.
